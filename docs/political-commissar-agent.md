# 政委 Agent（Meta Governance Layer）

## 为什么需要政委

在多 Agent 系统中，每个 Agent 只关注自己的局部任务，缺乏全局视角和质量把控。类比真实组织：

> 指挥员负责打仗，政委负责路线。

如果没有政委，Agent 系统会出现：
- Agent 为完成任务不择手段（编造数据、偷工减料）
- 多重 Agent 间错误信息互相放大
- 没有"局外人"来审视决策的合理性

## 设计原则

1. **不参与执行**——政委不写代码、不生成内容，只做审核
2. **关键节点介入**——在 Agent 输出前、跨 Agent 传递前、最终交付前进行抽检
3. **拥有否决权**——政委有权驳回低质量输出并要求重做
4. **战史记录**——每一次干预都记入战史库，用于后续优化

## 工作流程

```
Agent 完成任务
    ↓
政委接收输出
    ↓
政委检查清单：
  - 是否做过调研（有数据支撑？）
  - 是否存在明显幻觉？
  - Token 使用是否合理？
  - 是否偏离任务目标？
    ↓
通过 → 输出下游
驳回 → 打回重做 + 记录原因
```

## 实现参考

政委本质是一个独立的 LLM 调用，以审核 Prompt 驱动，不共享 Agent 的上下文窗口以避免被"污染"。设计上：

- 使用独立的 System Prompt
- 有明确的审核 checklist
- 输出格式标准化（通过/驳回 + 理由）

## 成本控制

政委的每次调用都会增加 Token 消耗。建议：
- 简单任务：跳过或 10% 抽检
- 中等复杂度任务：50% 抽检
- 高价值/高风险任务：100% 审核
